# MagicWindows Web — Design

**Date:** 2026-04-30
**Status:** Draft → ready for plan

## Goal

Ship a public website at `magicwindows.mindvisionstudio.com` that lets non-technical Windows users download a pre-built layout DLL + minimal installer, without needing the desktop app. The desktop app remains the full-featured experience (modifier toggles, F12 remap, health check, auto-detect).

## Non-Goals

- Replace the desktop app. Web is acquisition + lightweight install path only.
- Server-side DLL generation. `kbdutool.exe` is Windows-only and stays in the local release pipeline.
- User accounts, custom layouts, paid features.
- French/multilingual UI for V1 (English only; the desktop remains bilingual).
- Code signing for V1 (mitigated via `.cmd` wrapper + clear README).

## Architecture

### Hosting

- **Static site:** Cloudflare Pages, custom domain `magicwindows.mindvisionstudio.com`.
- **Downloadable ZIPs:** Cloudflare R2 bucket, public read access. URLs are versioned and immutable.
- **No Workers** in V1. Pages serves HTML/CSS/JS, R2 serves ZIPs over HTTPS directly.

### Stack

- Svelte 5 + Vite + TypeScript (consistent with desktop frontend).
- Pure SPA, no SSR. Output is static `dist/` deployed to Pages.
- No backend code, no database.

**Workspace topology:**

- Root `package.json` declares `"workspaces": ["packages/*", "web"]`.
- Desktop (`./` package, name: `magicwindows`) keeps its current scripts and depends on `@magicwindows/keyboard-visual` via workspace protocol.
- `packages/keyboard-visual` is published name `@magicwindows/keyboard-visual` (private, not pushed to npm).
- `web` package depends on `@magicwindows/keyboard-visual` and on `../layouts/*.json` via Vite's relative import (no extra package).

The `KeyboardVisual` component currently in `src/components/KeyboardVisual.svelte` is moved to `packages/keyboard-visual/src/KeyboardVisual.svelte`. The desktop's import is updated. The component must accept its layout data via props (it already does — verify before extraction).

## Components

### Web app (`web/`)

A 3-route SPA driven by hash-based routing (no history API needed for a static host).

- **`#/` — Home (Hero)**
  Headline: "Apple keyboard layouts on Windows"
  Subhead (1-2 lines): the same Shift+Enter / @ / # / éàç problem the desktop solves, framed for users who arrive cold.
  Primary CTA: "Choose your layout" → scrolls to layout grid.
  Layout grid: 6 cards (FR AZERTY, US, UK, DE, ES, IT), each card → `#/preview/{id}`.
  Secondary CTA in footer: "Want auto-detect, modifier toggles, F12 remap?" → `#/desktop`.

- **`#/preview/:layoutId`** — Interactive preview
  Header: layout name, switcher to flip between the 6 layouts in place.
  Body: `<KeyboardVisual layout={layouts[id]} />` (the shared component). Hover or tap a key → tooltip showing base / shift / altGr characters.
  Sticky bottom-right CTA: "Download for Windows" → triggers download from `manifest.downloads[id].url`.

- **`#/desktop`** — Desktop pitch
  Quick comparison table: web = layout install only, desktop = layout + modifier toggles + F12 + health check + auto-detect + uninstall UI.
  CTA: button linking to GitHub Releases page (`https://github.com/ObzCureDev/Magicwindows/releases/latest`).

### Shared package (`packages/keyboard-visual/`)

The existing `KeyboardVisual.svelte` extracted as a reusable component.

- Single export: `KeyboardVisual` (Svelte 5 component).
- Props: `layout: Layout`, `highlightedScancode?: string`, optional `onKeyHover?: (scancode) => void`.
- No app-specific logic. Must not import from `src/lib/stores.svelte.ts` or any desktop-only module. If the current component does, lift those concerns to the consumer.
- Owns its own CSS scoped to the component.

### Release script (`scripts/build-web-release.mjs`)

Local-only script run by the maintainer (you) at release time. Steps:

1. Read root `package.json` version (e.g. `0.3.0`).
2. For each layout in `layouts/*.json`:
   a. Run `klc-codegen` to produce `.klc`, then `kbdutool.exe -u <file>.klc -x` (x64) → `<dllname>.dll`. The exact invocation mirrors what `src-tauri/build.rs` already does — extract the shared logic into a Node-callable form, or shell out via `cargo run -p klc-codegen --bin <name>`.
   b. Compute SHA-256 of the DLL.
   c. Stage a temp dir with: `<dllname>.dll`, `Install-Layout.cmd`, `Install-Layout.ps1`, `Uninstall-Layout.cmd`, `Uninstall-Layout.ps1`, `README.txt`. The `.ps1` files are minimal versions adapted from the existing `scripts/Install-Layout.ps1` and `scripts/Uninstall-Layout.ps1`, scoped to one layout (the layoutId is inlined).
   d. ZIP the staging dir into `magicwindows-{layoutId}-{version}.zip`.
3. Upload each ZIP to R2: `wrangler r2 object put magicwindows-downloads/magicwindows-{layoutId}-{version}.zip --file <local>` (bucket name `magicwindows-downloads`, owned by Dan's Cloudflare account).
4. Write `web/public/manifest.json`:
   ```json
   {
     "version": "0.3.0",
     "generatedAt": "2026-04-30T12:00:00Z",
     "downloads": {
       "apple-fr-azerty": {
         "url": "https://magicwindows-downloads.r2.dev/magicwindows-apple-fr-azerty-0.3.0.zip",
         "size": 12345,
         "sha256": "abc..."
       },
       "apple-us-qwerty": { ... }
     }
   }
   ```
5. Print a summary table (layout / size / SHA / URL) and exit.

The script does NOT commit, push, or tag. The maintainer commits `web/public/manifest.json` after verifying the upload, then triggers the Pages deploy via a separate `git push`.

### ZIP contents (per layout)

- **`<dllname>.dll`** — e.g. `kbdmwfr.dll`. The compiled keyboard layout.
- **`Install-Layout.cmd`** — entrypoint, what the user double-clicks:
  ```cmd
  @echo off
  powershell -ExecutionPolicy Bypass -NoProfile -File "%~dp0Install-Layout.ps1"
  pause
  ```
- **`Install-Layout.ps1`** — does the actual install: copies DLL to `C:\Windows\System32`, registers the layout via `Set-WinUserLanguageList` (or direct registry write at `HKLM:\SYSTEM\CurrentControlSet\Control\Keyboard Layouts\<KLID>` matching what the desktop app does), prompts for restart. Self-elevates if not admin.
- **`Uninstall-Layout.cmd`** + **`Uninstall-Layout.ps1`** — symmetric reverse.
- **`README.txt`** — short, plain text:
  - "Double-click `Install-Layout.cmd`. Windows SmartScreen will warn — click **More info → Run anyway**. UAC will ask for admin rights — click **Yes**. Restart when asked."
  - Uninstall instructions.
  - Link: "Want auto-detect, modifier toggles, F12 remap? → magicwindows.mindvisionstudio.com/desktop"
  - Bug report link.

## Data Flow

### Build / release flow

```
maintainer machine                                           Cloudflare
─────────────────                                            ──────────
npm run release:web
  ├─ kbdutool → 6 DLLs                                       
  ├─ ZIP each (DLL + cmd + ps1 + readme)                     
  ├─ wrangler r2 put × 6  ────────────────────────────────►  R2 bucket
  └─ write web/public/manifest.json
maintainer commits manifest, pushes
  └─ Pages auto-builds web/  ─────────────────────────────►  Pages
```

### User flow

```
User                          Pages                         R2
────                          ─────                         ──
Visits magicwindows.mvs.com   Serves SPA shell
Picks layout (Preview page)   Loads manifest.json
Clicks "Download"             Redirects to manifest URL ─►  Streams ZIP
User extracts ZIP locally
User double-clicks .cmd
  ├─ SmartScreen warning, user clicks "Run anyway"
  ├─ UAC prompt, user clicks "Yes"
  ├─ PS1 copies DLL, writes registry
  └─ User restarts, layout active
```

## Error Handling

### Web side

- Manifest fetch failure: show fallback message "Downloads temporarily unavailable, please try again later." with a retry button. Manifest is bundled at build time anyway, so this should never fail in practice.
- Unknown layout in URL (`#/preview/foo`): redirect to `#/`.
- Missing manifest entry for a layout (build-time mismatch): hide that layout's download button, log a console warning.

### Install script side (`Install-Layout.ps1`)

- Not admin: self-elevate via `Start-Process powershell -Verb RunAs`. If user declines UAC, exit with clear message.
- DLL already exists in System32 with same name: prompt "An existing layout at this name was found. Overwrite? [Y/N]". Overwrite by default if user picked Yes via the cmd flow's `pause`.
- DLL copy fails (locked, antivirus): print error, suggest reboot and retry.
- Registry write fails: print error with the exact registry path so user can manually fix or report.
- Success: print "Installed. Restart your PC to activate the layout."

The script does NOT silently swallow errors. Every failure path prints what went wrong and suggests a next step.

## Testing

### Web app

- **Unit tests (vitest)** in `web/src/`:
  - Manifest parsing/validation (handles missing fields gracefully).
  - Hash router (correct route for `#/`, `#/preview/apple-fr-azerty`, unknown route).
  - Layout switcher state on Preview page.
- **Manual smoke test** before each release:
  - Open the deployed site, click each layout card, hover a few keys, click download.
  - Verify ZIP downloads and extracts cleanly.

### Shared package

- The existing tests for `KeyboardVisual` (if any) move with it. If there are none, add at least one snapshot/render test to lock the public API.

### Release script

- Run end-to-end on a clean checkout: produce 6 ZIPs, verify each contains the expected files, verify SHA-256 in manifest matches.
- Dry-run mode (`--dry-run`) skips R2 upload and manifest write — useful for CI sanity checks if we add CI later.

### Install script

- Tested manually on a fresh Windows 11 VM:
  - Double-click `.cmd`, accept SmartScreen + UAC, verify layout appears in Settings after reboot.
  - Run uninstall, verify layout removed.
- Existing `scripts/Install-Layout.ps1` is already battle-tested by the desktop flow; the simplified web variant is a strict subset (one layout, no toggles).

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| SmartScreen scares users away | Clear README + screenshots in the ZIP. The `.cmd` wrapper avoids ExecutionPolicy errors. Code signing deferred to V2 if conversion stinks. |
| User downloads stale ZIP after we ship a layout fix | Versioned URLs make new releases visible in the manifest. Site always advertises latest. Old URLs stay live (immutable) so nobody breaks, but home page shows current version. |
| `KeyboardVisual` extraction breaks the desktop | Move + update import in one commit, run `npm run check && npm run test && cargo test` before merging. Component already takes layout as input, so the boundary should be clean. |
| Workspace setup breaks Tauri build | `tauri build` runs `npm run build` which only builds the desktop's `src/`. Workspaces don't change this. Verify on a clean install: `rm -rf node_modules && npm install && npm run tauri build`. |
| R2 bucket misconfiguration leaves ZIPs private | Bucket is `public-bucket` policy with public-read enabled in dashboard. Smoke test: open the URL from a private browser window after first upload. |
| layouts JSON shape changes break the web | Both apps consume the same files. A schema-breaking change forces both to update — desirable, single source of truth. |
| Domain DNS not under Dan's control | `magicwindows.mindvisionstudio.com` requires a CNAME on `mindvisionstudio.com`. Pre-flight check before the implementation plan finalizes. |

## Future / Out of Scope

- Auto-detection on the web (browser API doesn't expose Windows keyboard ID — would need a small helper exe).
- Custom layout uploads.
- Code signing the `.ps1` (Authenticode OV cert ~$80/yr).
- French i18n on the site.
- Dark mode (the desktop app has it, web doesn't for V1).
- A/B testing of CTAs.
- Newsletter / mailing list capture.
- "Live demo" embedding the desktop app via WASM (not feasible — Tauri/Rust).
